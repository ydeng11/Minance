| Token | Value | Usage |
|-------|-------|-------|

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | #0a0a0a | Page background, primary surfaces, workflow canvas |
| Secondary (30%) | #171717 | Cards, inline alert fills, table heads, detail panels |
| Accent (10%) | #34d399 | Success toasts, focus rings, positive status icons, successful completion emphasis |
| Destructive | #f43f5e | Delete confirmations, destructive validation emphasis, destructive action buttons only |

Accent reserved for: success toasts, successful Import completion states, successful mapping/account-assignment confirmations, Recurrings save/evaluate/lifecycle success confirmations, and focus-visible rings. Do not use accent green for generic page decoration, neutral controls, or passive informational copy.

Additional semantic guidance:
- Error and recovery text should stay in the existing neutral-surface family with rose text or icons only where the state is actually blocking.
- Inline alerts should remain dark-surface anchored (`border-neutral-800`, `bg-neutral-900/60`) so they read as persistent guidance rather than celebratory banners.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Process Transactions |
| Empty state heading | No recurring rules yet |
| Empty state body | Create a rule above to track a repeating charge or deposit, then run evaluation to link matching transactions. |
| Error state | {Action} couldn’t be completed. Nothing changed. {Specific next step}. |
| Destructive confirmation | Delete recurring rule: Delete this rule? This removes the rule only. Linked transactions stay unchanged. |

Microcopy rules:
- Name the failed action directly: `Import analysis couldn’t finish.`, `Mapping couldn’t be saved.`, `Recurring rule couldn’t be updated.`
- State impact before advice: `Nothing changed.` or `Your current draft is still here.` when true.
- End with one concrete next step: `Check the required fields and try again.`, `Choose an account and retry.`, `Reconnect the import mapping, then commit again.`
- Prefer remediation phrasing over raw API text unless `ApiError` already provides actionable language.

## Feedback Surface Contract

### Visual priority

- Import: the first focal point is the active workflow-status region directly below the page header; if there is no blocking inline alert, the primary import action area for the current step should lead the eye before row tables or secondary summaries.
- Recurrings: the first focal point is the active action surface, meaning the create/edit rule form or the blocking inline alert when present; totals and suggestions remain secondary context, not the page's visual anchor.

### Success confirmations

Use `sonner` toasts for completed critical actions only. Keep the existing dark toaster (`top-center`, dark theme, close button, rich colors) from `AppProviders`.

| Surface | Action | Success copy |
|---------|--------|--------------|
| Import | Analyze | Import analyzed. Review the preview before saving mapping. |
| Import | Save mapping | Mapping saved. Continue to account review. |
| Import | Assign import account | Import account assigned. Review any remaining unmatched rows. |
| Import | Map reconciliation account | Reconciliation account mapped. Check the remaining issues before commit. |
| Import | Commit | Import committed. Review imported transactions if anything looks off. |
| Recurrings | Create | Recurring rule created. |
| Recurrings | Save edits | Recurring rule updated. |
| Recurrings | Evaluate | Evaluation complete. {N} matches found. |
| Recurrings | Pause | Recurring rule paused. |
| Recurrings | Resume | Recurring rule resumed. |
| Recurrings | Archive | Recurring rule archived. |
| Recurrings | Delete | Recurring rule deleted. |

Toast rules:
- Auto-dismiss in 4 seconds.
- Use one sentence, max two clauses.
- Do not duplicate the same success in the inline alert region.

### Inline alerts

Keep a stable page-level inline alert region directly below the page header on Import and Recurrings using the existing `data-testid="global-message"` anchor.

Inline alerts are required for:
- Analyze/load failures.
- Mapping save failures.
- Account assignment failures.
- Reconciliation mapping failures.
- Import commit failures.
- Recurrings load/detail failures.
- Recurrings form submit failures that affect the whole action.

Inline alert rules:
- Persistent until replaced by a newer blocking issue or explicitly cleared by the next successful workflow step.
- Include what failed, what was affected, and the next step.
- Use icon + text treatment when implementation touches markup; otherwise preserve the current panel styling and upgrade only the copy.

### Field-level validation

Recurrings create and edit forms must show field-level validation for obvious input problems before submit:
- Missing rule name.
- Amount missing, non-numeric, or `<= 0`.

Validation rules:
- Render feedback adjacent to the offending field, not only in the global alert.
- Use 12px label-sized text with destructive color.
- Preserve entered input values after validation failure.
- Keep page-level inline alerts for cross-field or request-level failures only.

## Interaction Contract

- Keep success feedback transient and non-blocking; users should continue working without dismissing a panel.
- Keep error feedback anchored to the workflow region the user is actively using.
- Do not convert recovery-oriented errors into toasts.
- Preserve draft state on failed actions wherever the current UI already retains it.
- When an action is in flight, keep the current disabled-state pattern: `disabled:opacity-60` and `disabled:cursor-not-allowed`.
- When a button is loading, pair the existing `Loader2` spinner with a verb phrase: `Processing via AI...`, `Saving mapping...`, `Evaluating links...`.
- Destructive actions stay inline and require explicit confirmation in place, not a modal.

Page-specific interaction notes:
- Import: success toasts replace reducer notice copy for routine success states, but reconciliation/account-assignment guidance may remain inline when it informs the next decision.
- Recurrings: global inline message region becomes error-only and persistent-guidance-only; routine success messages move to toasts.
