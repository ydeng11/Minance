## Phase Details

### Phase 1: Filter UX Foundation
**Goal:** Users can filter transactions intuitively with clear visibility into active filters
**Depends on:** Nothing (first phase)
**Requirements:** FILT-01, FILT-02, FILT-03, FILT-04, FILT-08
**Success Criteria** (what must be TRUE):
  1. User sees a single search input that finds matches across merchant, description, notes, and tags
  2. User can see all active filters as removable badges outside the filter panel
  3. User can share a filtered view by copying the URL (filters are in URL state)
  4. User experiences responsive search with no excessive API calls (debounced input)
  5. User can clear all filters with a single action
**Plans:** 3 plans
Plans:
- [x] 01-01-PLAN.md — Foundation: Dependencies & ActiveFilterBadges component
- [x] 01-02-PLAN.md — Explorer: Search consolidation & badge integration
- [x] 01-03-PLAN.md — Transactions: Debounced search, Clear all, badges, typo fix
**UI hint:** yes

### Phase 2: Visual Consistency & Table Polish
**Goal:** Users can scan and understand transaction data quickly with clear visual hierarchy
**Depends on:** Phase 1
**Requirements:** FILT-05, FILT-06, FILT-07, VISL-01, VISL-02, VISL-03, VISL-04, VISL-05, UFBK-03, UFBK-04
**Success Criteria** (what must be TRUE):
  1. User can quickly scan amounts in tables (right-aligned, green for income, red for expenses)
  2. User always knows when data is loading (spinners, skeletons)
  3. User sees helpful guidance when no transactions match their filters
  4. User can select multiple categories or accounts to filter by
  5. User can use date range presets for quick filtering
**Plans:** 4/4 plans complete
Plans:
- [x] 02-01-PLAN.md — Date range presets: Extend RANGE_OPTIONS (FILT-05)
- [x] 02-02-PLAN.md — Table polish: Amount colors, sticky headers, empty state (VISL-01, VISL-05, VISL-03)
- [x] 02-03-PLAN.md — Loading states: Skeleton rows, button spinners (VISL-02, UFBK-03)
- [x] 02-04-PLAN.md — Consistency audit: Spacing, disabled states (VISL-04, UFBK-04)
**Notes:**
- FILT-06 and FILT-07 already implemented — MultiSelectField used for categories/accounts in both Transactions and Explorer
**UI hint:** yes

### Phase 3: Interaction Patterns
**Goal:** Users can perform actions confidently with immediate, predictable feedback
**Depends on:** Phase 2
**Requirements:** INTX-01, INTX-02, INTX-03, INTX-04, INTX-05, INTX-06
**Success Criteria** (what must be TRUE):
  1. User sees inline validation errors immediately on blur with actionable messages
  2. User never loses entered data on validation error (input preservation)
  3. User receives toast confirmations for important actions (save, delete, import)
  4. User can undo destructive actions within 5 seconds via toast notification
  5. User can quickly navigate and perform actions via command palette (cmd+k)
  6. User sees actions complete instantly with rollback on error (optimistic updates)
**Plans:** 4/4 plans complete
Plans:
- [x] 03-01-PLAN.md — Toast infrastructure (Sonner) + Transactions migration (INTX-03)
- [x] 03-02-PLAN.md — Inline validation blur + input preservation (INTX-01, INTX-02)
- [x] 03-03-PLAN.md — Optimistic updates + undo via toast (INTX-04, INTX-05)
- [x] 03-04-PLAN.md — Command palette cmd+k (INTX-06)
**UI hint:** yes

### Phase 4: User Feedback & Error Handling
**Goal:** Users always understand what happened and what to do next
**Depends on:** Phase 3
**Requirements:** UFBK-01, UFBK-02
**Success Criteria** (what must be TRUE):
  1. User sees clear error messages with specific recovery actions (not generic errors)
  2. User receives success confirmations for critical actions (save, delete, import)
**Plans:** 2/2 plans complete
Plans:
- [x] 04-01-PLAN.md — Import feedback routing: shared recovery-copy helper, success toasts, inline recovery alerts, and Import E2E migration
- [x] 04-02-PLAN.md — Recurrings validation split: field-level errors, success toasts, and parity E2E migration
**UI hint:** yes
